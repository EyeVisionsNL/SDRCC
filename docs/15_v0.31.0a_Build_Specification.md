# v0.31.0a Build Specification

## Release name

`v0.31.0a - Receiver Manager Multi-Reservation Foundation`

## Functional goal

Allow independent runtime reservations for SDR1 and SDR2 without changing current mission execution.

## Acceptance criteria

1. SDR1 and SDR2 can be reserved independently.
2. The same receiver cannot be reserved twice by different mission keys.
3. Activation and release affect only the matching reservation.
4. Existing consumers continue receiving singular compatibility fields.
5. Existing legacy state migrates automatically.
6. Preflight checks only the selected receiver.
7. Stale reservation reconciliation never releases the other SDR.
8. Service restart preserves valid reservations.
9. Installation and rollback are automated.
10. No Git commit is performed by the package.

## Test order

1. Run package syntax validation.
2. Install package.
3. Restart `sdrcc.service`.
4. Check service status.
5. Check `/api/receiver-manager`.
6. Run idle reservation validation script.
7. Confirm both SDRs remain in their default roles after the test.
8. Perform current single METEOR workflow validation later.
9. Commit only after user approval.

## Explicit exclusions

- parallel SatDump processes;
- runtime registry;
- multiple Mission Engine instances;
- scheduler allocator;
- Mission Control UI redesign;
- default role redesign;
- plugin framework changes.
