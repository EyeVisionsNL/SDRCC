# v0.31.0a Receiver Manager Multi-Reservation Impact Analysis

## Scope

This document defines the exact impact of the first Epic 1 code change.

The goal is limited to making SDR1 and SDR2 independently reservable while preserving the current single-mission behaviour.

No Mission Engine, scheduler or UI redesign belongs in this version.

## Confirmed current state

`core/receiver_manager.py` currently persists one station-wide reservation:

```json
{
  "reservation": null,
  "last_release": null
}
```

This means a reservation on SDR1 also makes SDR2 unavailable.

The current public interface is:

```python
get_status()
is_available(receiver_id, mission_key=None)
reserve(receiver_id, mission_key, mission_id=None, reason=...)
activate(mission_key, mission_id=None)
release(mission_key=None, detail=...)
```

Existing consumers expect the singular compatibility fields:

```text
reservation
last_release
available
configured_receiver
```

## Required code impact

### 1. `core/receiver_manager.py`

Mandatory change.

New persisted structure:

```json
{
  "reservations": {
    "sdr1": {},
    "sdr2": {}
  },
  "last_releases": {
    "sdr1": {},
    "sdr2": {}
  }
}
```

Required behaviour:

- reservations are keyed by receiver ID;
- SDR1 and SDR2 can be reserved by different mission keys;
- a receiver can have only one reservation;
- one mission key cannot silently move to another receiver;
- activation targets the matching mission reservation;
- release targets the matching mission reservation;
- state writes remain atomic;
- old state files are migrated during load;
- no destructive manual migration is required.

New fields returned by `get_status()`:

```text
receivers
reservations
last_releases
available_receivers
```

Compatibility fields retained:

```text
configured_receiver
reservation
last_release
available
```

Compatibility meaning:

- `reservation`: reservation for the configured dynamic/weather receiver;
- `last_release`: last release for that configured receiver;
- `available`: availability of that configured receiver.

This preserves current dashboard behaviour while exposing the new model.

### 2. `core/mission_preflight.py`

Small compatibility change recommended.

Current logic reads the singular `reservation` field and checks whether it belongs to the dynamic receiver.

It should instead use:

```python
receiver_manager.is_available(
    device["id"],
    mission_key=<optional current mission key>
)
```

For v0.31.0a, when no mission key is available, it may simply test the selected receiver.

This prevents preflight from being blocked by an unrelated reservation on the other SDR.

### 3. `core/mission_queue.py`

Small compatibility change recommended.

Current queue display reads the singular reservation.

It should look up the reservation for the configured receiver from:

```python
status["reservations"].get(receiver_id)
```

This keeps queue display correct when the other SDR is reserved.

No queue allocation redesign belongs in this version.

### 4. `dashboard/app.py`

Minimal compatibility change required.

`get_reconciled_receiver_manager_status()` currently assumes one reservation and may release it globally.

It must only reconcile the reservation that belongs to the current autopilot/weather mission.

Safe rule:

- determine the configured dynamic receiver;
- inspect only that receiver reservation;
- release only when no current mission runtime owns that reservation;
- never release a reservation belonging to the other SDR.

The receiver assignment guard should block assignment changes only when the selected receiver is reserved, not merely because any receiver is reserved.

### 5. `core/mission_operations.py`

No mandatory functional change.

It can continue embedding `receiver_manager.get_status()` because the response remains backwards compatible.

Optional only:

- expose new multi-receiver fields automatically through the existing snapshot.

## Public API compatibility

Existing endpoint:

```text
GET /api/receiver-manager
```

must continue returning the current fields.

It may additionally return:

```json
{
  "receivers": {},
  "reservations": {},
  "last_releases": {},
  "available_receivers": []
}
```

No new write endpoint is required for this foundation version.

## State migration

Old state:

```json
{
  "reservation": {
    "receiver_id": "sdr1",
    "mission_key": "..."
  },
  "last_release": {}
}
```

Migration on load:

```json
{
  "reservations": {
    "sdr1": {
      "receiver_id": "sdr1",
      "mission_key": "..."
    }
  },
  "last_releases": {
    "sdr1": {}
  }
}
```

The old keys should not be written back after successful migration.

Rollback warning:

A rollback to the old Receiver Manager cannot interpret the new state format. Therefore the rollback script must:

1. back up the current state file;
2. restore the old Python files;
3. convert the configured receiver reservation back to the singular legacy format;
4. preserve the complete new state backup for manual recovery.

## Validation matrix

### State migration

- no state file;
- valid old state file;
- valid new state file;
- malformed JSON;
- unknown receiver ID.

### Independent reservation

- reserve SDR1 with mission A;
- verify SDR2 remains available;
- reserve SDR2 with mission B;
- verify both reservations exist;
- reject mission C on SDR1;
- reject mission A attempting to move to SDR2.

### Activation

- activate mission A;
- verify only SDR1 becomes ACTIVE;
- verify SDR2 remains RESERVED;
- activate mission B;
- verify both are ACTIVE independently.

### Release

- release mission A;
- verify SDR1 is free;
- verify SDR2 remains active;
- release mission B;
- verify both are free;
- reject release with wrong mission key.

### Compatibility

- `reservation` reflects configured receiver only;
- `available` reflects configured receiver only;
- current Mission Queue renders;
- preflight is not blocked by reservation on the other SDR;
- `/api/receiver-manager` returns HTTP 200;
- no JavaScript errors from additional response fields.

## Go/no-go decision

GO.

The change is small enough for one isolated package, provided:

- legacy state migration is included;
- compatibility fields remain;
- stale-reservation reconciliation is receiver-specific;
- rollback includes state conversion;
- no Mission Engine or UI refactor is added.
